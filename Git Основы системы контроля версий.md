**Git** — это **распределенная система контроля версий (DVCS)**. Ее главная задача — отслеживать изменения в файлах проекта с течением времени, позволяя множеству разработчиков эффективно работать вместе.

> **Основная идея**: Git делает "снимки" (snapshots) вашего проекта каждый раз, когда вы фиксируете изменения. Это позволяет вам возвращаться к любой предыдущей версии, анализировать историю изменений и безопасно работать над новыми функциями, не ломая основной, рабочий код.

**Ключевой факт**: В отличие от централизованных систем (как SVN), где главная копия проекта хранится на одном сервере, в Git у **каждого разработчика** есть **полная копия всей истории** проекта. Это делает систему очень быстрой и надежной.

---

### Основной рабочий процесс (локально)

Представьте, что у вас есть три "зоны":

1. **Рабочая директория (Working Directory)**: Файлы, которые вы сейчас редактируете.
    
2. **Область подготовленных файлов (Staging Area / Index)**: Промежуточная зона. Сюда вы добавляете файлы, которые готовы к "фотографированию". Это позволяет создавать осмысленные, атомарные коммиты.
    
3. **Локальный репозиторий (.git)**: База данных Git, где хранятся все ваши "снимки" (коммиты).
    

**Цикл работы:**

1. Вы изменяете файлы в **рабочей директории**.
    
2. Командой git add вы добавляете готовые изменения в **staging area**.
    
3. Командой git commit вы делаете "снимок" staging area и сохраняете его в **локальный репозиторий**.

### Ключевые команды и их назначение

#### git clone [url]

- **Что делает?** Клонирует (копирует) удаленный репозиторий (например, с GitHub) на ваш локальный компьютер. Это ваша отправная точка.
    

#### git status

- **Что делает?** Показывает текущее состояние репозитория: какие файлы изменены, какие находятся в staging area, а какие еще не отслеживаются. **Это ваша главная команда для ориентации в пространстве.**
    

#### git add [file_name] или git add .

- **Что делает?** Добавляет указанный файл (или все файлы с . ) из рабочей директории в staging area.
    

#### git commit -m "Your descriptive message"

- **Что делает?** Создает новый коммит (снимок) из файлов, находящихся в staging area.
    
- **Важно**: Сообщение коммита (-m) должно быть осмысленным! "Исправлен баг X", "Добавлена функция Y".
    

#### git log

- **Что делает?** Показывает историю коммитов. Вы увидите хеш (уникальный ID) каждого коммита, автора, дату и сообщение.
    

#### git push

- **Что делает?** Отправляет ваши локальные коммиты на удаленный репозиторий (например, GitHub), делая их доступными для других.
    

#### git pull

- **Что делает?** Забирает изменения с удаленного репозитория и пытается автоматически слить их с вашей текущей версией.
    
- **Интересный факт**: git pull — это на самом деле комбинация двух команд: git fetch (которая только скачивает изменения, но не применяет их) и git merge (которая сливает скачанные изменения).

## Ветвление (Branching) и Pull Requests

Это ядро совместной работы.

### [[Ветви (Branches)]]

**Ветка** — это независимая линия разработки. Основная ветка обычно называется main или master и содержит стабильный, рабочий код.

- **Зачем?** Чтобы работать над новой функцией или исправлять баг, вы создаете новую ветку. В ней вы можете делать что угодно, не боясь сломать основной код. Когда работа завершена, вы сливаете (merge) вашу ветку обратно в main.
    

#### git branch [branch_name]

- **Что делает?** Создает новую ветку.
    

#### git checkout [branch_name]

- **Что делает?** Переключается на указанную ветку.
    

#### git checkout -b [branch_name]

- **Что делает?** "Комбо": создает новую ветку и сразу же переключается на нее. Это самая частая команда для начала новой работы.
    

#### git merge [branch_name]

- **Что делает?** Вливает изменения из указанной ветки в вашу текущую ветку.
    

### [[Pull Requests (PR) / Merge Requests (MR)]]

Это механизм для **предложения** ваших изменений в основную ветку на удаленном репозитории (GitHub, GitLab, etc).

**Процесс:**

1. Вы создаете новую ветку локально (feature-x).
    
2. Делаете в ней коммиты.
    
3. Отправляете вашу ветку на удаленный репозиторий (git push origin feature-x).
    
4. На сайте (GitHub) вы создаете **Pull Request**, предлагая слить вашу ветку feature-x в ветку main.
    
5. **Зачем?** Это точка для **обсуждения и код-ревью**. Ваши коллеги могут посмотреть на ваши изменения, оставить комментарии, а автоматические тесты (CI/CD) могут проверить код на ошибки. После одобрения PR сливается в main.

### Популярные вопросы на собеседовании

- **"Что такое Git?"**
    
    - Ответ: Это распределенная система контроля версий...
        
- **"В чем разница между git pull и git fetch?"**
    
    - Ответ: git fetch только загружает удаленные изменения, но не применяет их. git pull делает fetch, а затем merge. fetch безопаснее, так как позволяет посмотреть на изменения до их слияния.
        
- **"Что такое Staging Area?"**
    
    - Ответ: Это промежуточная область между рабочей директорией и репозиторием. Она позволяет группировать изменения и создавать логически завершенные коммиты.
        
- **"Зачем нужны ветки?"**
    
    - Ответ: Для изоляции работы. Они позволяют вести параллельную разработку нескольких фич или исправлений, не мешая друг другу и не затрагивая стабильную версию кода в основной ветке.
        
- **"Что такое Pull Request?"**
    
    - Ответ: Это механизм для код-ревью и обсуждения изменений перед их вливанием в основную ветку проекта.
